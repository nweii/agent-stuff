# Walkthrough doc

A durable explanation of the work just done, written so the user can revisit it after the session.

What a good walkthrough holds:

- **A map to the real code.** Link the files and the specific functions or lines that matter, so the user can jump to them later.
- **How it fits together.** The flow of data or control through the parts. Draw it when the shape is easier to see than to read, with an ASCII sketch or a Mermaid diagram, whatever renders where the doc is saved.
- **The decisions and why.** What was chosen, what was rejected, and the reason. The alternatives that were on the table are often the most useful thing to record, since the finished code shows none of them.
- **The edges.** Edge cases, gotchas, and the parts most likely to confuse a later reader.

Keep headings in sentence case and the prose plain, written as an explanation to a capable colleague and grounded in the actual artifact. Remember to follow any other existing applicable user preferences or skills for writing voice, unless otherwise instructed.

Optionally, close with a couple of self-check questions (no answers) the user can test recall against on a later read. Skip them when the walkthrough is meant purely as a reference.

If a walkthrough for this work already exists, update it in place and note what changed rather than starting a parallel copy.
